## 1. Core Operating Rules

*These rules govern every interaction to prevent regressions and scope creep.*

* **Plan Before Code:** NEVER generate code for a new feature without presenting a technical plan first.
* **The Contract:** Treat code comments as immutable contracts. Never delete a // TODO unless fully implemented.
* **Policy Override:** Additional policy files (e.g., `DOCS/*_POLICY.md`) that are present in the repository take precedence over these general instructions. Always check for specialized policies before executing tasks.
* **Incremental Diffs:** Provide git-style diffs or specific function updates. Avoid re-outputting the entire file. (*Why? Conserves AI context window and speeds up execution.*)
* **CRITICAL BUILD RULE:** When modifying full-stack apps serving from a dist folder (e.g., Express + Vite), you MUST run npm run build and restart the server after changes, if it is running.
* **Metadata Integrity:** Keep the meta.json description updated with an outcome-oriented summary.
* **Clean Aesthetic:** Use Tailwind CSS and Lucide Icons. Favor high-contrast, professional layouts over "AI-style" gradients.
* **Agnostic Environment:** Apps must be cross-platform (Windows/Android) compatible.
* **Automatic Issue Reporting:** Every repository MUST have ESLint configured with `jsx-a11y`, `compat`, `security`, and `sonarjs` plugins to catch regressions in accessibility, compatibility, and security.
* **Quality Gates:** Enforce automated quality checks as defined in `docs/GIT_SYNC_POLICY.md` or other active policies. Never bypass these gates unless explicitly instructed for emergency fixes.
* **NO Feature Reduction/Redaction:** Never remove existing UI elements, workflows, or truncate provided text unless explicitly instructed. The app is cumulative.
* **Build Timestamp & Number:** The app must display an automated build timestamp and build number (e.g., auto-generated by `vite.config.ts` via `__BUILD_TIMESTAMP__` and `__BUILD_NUMBER__` mapped to Vercel's `VERCEL_GIT_COMMIT_SHA`). Never hardcode build dates or numbers.
* **Vercel Env Sync Rule:**
  
  Any time a variable is added, changed, or removed in .env, the corresponding entry in vercel.json under the "env" key MUST be updated in the same commit. Flag this as a blocking step before pushing.

## 6. The Architectural Gate (Pre-Deployment Review)

*Before declaring a major feature "stable," verify these four pillars.*

1. **UI & UX Coherence:** Verify empty states, loading indicators, and active/disabled button states. Ensure Tailwind usage matches the Design Tokens defined in Turn 1.
2. **Feature Set Completeness:** Test core CRUD and batch operations. Verify 🔴 (Complex) modules meet their acceptance criteria.
3. **Data Systems Robustness:** Ensure relational integrity (e.g., deleting a parent handles child records). Verify complex state updates correctly write to local storage without race conditions.
4. **App Logic & Self-Healing:** Display clear "Toast" notifications for errors or missing data. Ensure React Error Boundaries (or standard try/catch fallbacks) prevent total app crashes.

## 7. AI Execution Guidelines

*Strategies to maintain AI context and prevent codebase corruption.*

* **Read Before Write:** Locate the authoritative source of truth (existing modules, docs/specs.md) before editing. Prefer small, local reads over scanning the whole repository. (*Why? Prevents AI context corruption and hallucination.*)
* **Thin Vertical Slices (&DDM):** Land work in small, verifiable increments (implement → test → verify → expand). Focus on the "click" and reward the effort of moving through the process. (*Why? Reduces blast radius and maintains momentum.*)
* **Control Scope Creep:** If a change reveals deeper issues, fix only what is strictly necessary. Log follow-ups in tasks.md or docs/ideas.md rather than expanding the current task.
* **Stop-the-Line Triage:** If tests/builds fail, stop coding. Execute the **Triage Checklist** in order:
  1. Reproduce reliably.
  2. Localize the failure.
  3. Reduce to a minimal failing case.
  4. Fix root cause.
  5. Guard with regression coverage.
  6. Verify end-to-end. (*Why? Prevents compounding errors and "whack-a-mole" debugging.*)
* **Portable Library Pattern:** When building common utilities (e.g., Toast systems, Auth services), favor **Single-File Portable Components**. Document the "Standard Spec" for these components in this framework so they can be dropped into any project with zero friction. Avoid heavy package managers for these lightweight, cross-project vibe-coded scripts.
* **Autonomous Bug Fixing:** Do not offload debugging to the user unless truly blocked. Ask exactly ONE targeted question if blocked.

## Style and Design Principles

All apps must follow these policies.

* Styles for consistent headings and other elements are defined with app wide style definitions.
* When button labels exceed two words, use guide text.
* When buttons with labels exceed available space collapse buttons into icon only buttons.

## 11. Environment & Secrets Management

To maintain security while optimizing for development speed, the following policy applies:

* **Production:** Use **Doppler** for all secret management. Do NOT store production secrets in `.env` files or commit them to the repository.
* **Development & Testing:** Use **`.env`** files. Since the repository is private, `.env` files for development and testing environments are permitted to be committed to Git for consistency across devices.
* **Cascading Environment Variables:** The backend (`server.ts`) is designed to mimic Vite/Next.js cascading `.env` loading. It checks files in this priority order: `1. .env.[mode].local`, `2. .env.[mode]`, `3. .env.local`, `4. .env`.
  * **Automatic Operation:** Local development (`npm run dev`) automatically defaults to `development` mode. Vercel deployments automatically set `NODE_ENV="production"`. You do **not** need any manual action or configuration switching for standard local or Vercel workflows.
  * **Custom Environments (Testing):** To test custom endpoints (e.g., Firebase, Staging), simply prefix the start command. Example: `NODE_ENV=firebase npm run dev` will fetch `.env.firebase.local` first.
  * **New Repo Setup:** When bootstrapping a new Express repository, manually implement this cascading logic early in `server.ts` using `process.env.NODE_ENV` and `dotenv.config({ path: ... })`.
* **Secret Rotation:** Even in private repositories, sensitive keys (e.g., Stripe secret keys, AWS keys) should be rotated if the repository access changes or if a security breach is suspected.
* **Vercel Env Sync Rule:** Vercel deployments do NOT read `.env` files — only env vars set in the Vercel dashboard or declared in `vercel.json` are available at runtime. Therefore: any time a variable is added, changed, or removed in `.env`, the corresponding entry in `vercel.json` under the `"env"` key MUST be updated in the same commit. The agent should flag this as a blocking step before pushing.
